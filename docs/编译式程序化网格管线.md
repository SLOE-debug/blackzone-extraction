我同意你的判断，而且我觉得问题不只是“调用树还能缩短”，而是当前的**抽象边界放在了 Writer 层，位置偏低**。

你现在复用了 `TriangleMeshWriter`、TypedArray 和渲染适配器，但 Vanguard、蜘蛛在每一帧仍然通过 `GeometrySource.write()` 重新“讲述一遍网格是怎么组成的”：遍历 Patch、判断 Triangle/Quad、调用 Tessellator、重新走 Writer。`writer.reset(false)` 虽然不再写 Index，但拓扑控制流仍完整执行。大厅因为只运行一次，所以问题不明显。

真正适合抽象的是：

> **把“生成网格”拆成一次性的拓扑编译，以及每帧的顶点求值。**

---

# 一、建议把网格明确分成三种表示

这是整个重构里最重要的一步。

## 1. Control Geometry：领域控制数据

这是模型真正的形状定义，不直接上传 GPU。

Vanguard：

* 共享笼顶点
* 骨骼索引和权重
* Triangle / Quad / FacetedQuad
* Surface 语义
* 绑定姿态

蜘蛛：

* 腿的 Bézier 控制参数
* Tube 的段数和径向采样
* 椭球参数
* 眼睛、液体等部件定义

大厅：

* Grid Recipe
* SurfaceFrame
* Deformer
* 径向观察窗定义

这一层允许高度领域化，不要强行统一。

---

## 2. Compiled Mesh Plan：编译后的固定拓扑计划

这是目前最缺的一层。

它只在初始化时产生一次，保存所有不随帧变化的信息：

```ts
interface MeshPlan {
  readonly vertexCount: number;
  readonly indexCount: number;

  readonly indices: Uint16Array | Uint32Array;

  // 最终渲染顶点属于哪个语义区域
  readonly semanticIds: Uint8Array;

  // 可选：所属材质或 Surface
  readonly materialSlots?: Uint8Array;
}
```

对于 Flat Shading 的 Vanguard，还可以有专门计划：

```ts
interface FlatCageMeshPlan extends MeshPlan {
  readonly controlVertexCount: number;

  // 每个最终渲染顶点对应哪个共享笼顶点
  readonly renderToControlVertex: Uint16Array;

  // 每个三角形对应的语义 Surface
  readonly faceSemanticIds: Uint8Array;
}
```

假设笼中一个 Quad 最终拆成两个三角形，那么这个拆分只在编译时做一次。

之后每帧不再调用：

```text
appendPatch()
├─ Triangle
├─ Quad
└─ FacetedQuad
```

每帧只执行：

```text
共享笼顶点蒙皮
→ 按 renderToControlVertex 展开位置
→ 每三个顶点计算一个面法线
```

---

## 3. Runtime Vertex Streams：运行时顶点流

只保存真正会改变、会上传的属性：

```ts
interface VertexStreams {
  readonly positions: Float32Array;
  readonly normals: Float32Array;
  readonly colors?: Uint8Array | Float32Array;
}
```

再明确每个流的变化频率：

| 数据              | 生命周期            |
| --------------- | --------------- |
| Index           | 永久静态            |
| 顶点映射、参数坐标、语义 ID | 永久静态            |
| Position        | 姿态变化时           |
| Normal          | Position 变化时    |
| Color           | 调色板、受击、死亡等状态变化时 |
| Bounds          | Position 变化时    |

这样，你的抽象不再是“所有东西都能通过 Writer 写”，而是：

> **静态 Plan + 动态 Streams + 按需 Evaluator。**

---

# 二、当前最值得拆开的四个职责

目前的 `GeometrySource.write(writer)` 实际上承担了四件事：

1. 决定拓扑；
2. 计算顶点位置；
3. 计算法线；
4. 将结果顺序写进缓冲区。

这让静态大厅和动态角色看起来可以共用同一个接口，但它们其实只有最终输出格式相同，生命周期完全不同。

建议拆成：

```ts
interface MeshCompiler<TRecipe, TPlan extends MeshPlan> {
  compile(recipe: TRecipe): TPlan;
}

interface MeshEvaluator<TState, TPlan extends MeshPlan> {
  evaluate(
    state: TState,
    plan: TPlan,
    streams: VertexStreams,
    dirty: MeshDirty,
  ): MeshDirty;
}
```

Dirty Mask 可以非常简单：

```ts
export const enum MeshDirty {
  None     = 0,
  Position = 1 << 0,
  Normal   = 1 << 1,
  Color    = 1 << 2,
  Bounds   = 1 << 3,

  Geometry = Position | Normal | Bounds,
  All      = Geometry | Color,
}
```

Renderer 最终只负责：

```ts
class DynamicMeshSurface<TState, TPlan extends MeshPlan> {
  update(state: TState, requested: MeshDirty): void {
    const changed = this.evaluator.evaluate(
      state,
      this.plan,
      this.streams,
      requested,
    );

    this.gpuMesh.upload(this.streams, changed);
  }
}
```

这会让 `FixedTopologyBatchRenderer` 从“生成几何 + 管理缓冲 + 更新颜色 + 上传 GPU”的大编排器，收缩成真正的运行时容器。

---

# 三、Vanguard 最适合先改

Vanguard 是最容易获得明显收益的部分，因为它已经有很清晰的共享笼顶点。

## 当前路径

```text
每帧
→ deformVertices()
→ appendPatch()
→ 判断 Triangle / Quad / FacetedQuad
→ appendVanguardTriangle()
→ 写 Position / Normal
→ VertexShading.update()
→ 上传 Position / Normal / Color
```

这里真正每帧需要变化的只有：

* 蒙皮后的控制点位置；
* Flat Normal；
* 极少数情况下的 Color。

## 改成编译模型

初始化：

```text
Vanguard Cage
→ compileFlatCage()
   ├─ 展开 Triangle / Quad / FacetedQuad
   ├─ 生成 renderToControlVertex
   ├─ 生成固定 Index
   ├─ 生成 faceSemanticIds
   └─ 生成 vertexSemanticIds
→ VanguardMeshPlan
```

每帧：

```text
骨骼矩阵
→ skinControlVertices()
→ expandFlatPositions()
→ Core writeSequentialFlatNormals()
→ upload Position + Normal
```

大致代码形态：

```ts
function evaluateVanguardGeometry(
  plan: FlatCageMeshPlan,
  skin: VanguardSkinState,
  controlPositions: Float32Array,
  streams: VertexStreams,
): void {
  skinTwoWeights(
    skin.boneMatrices,
    plan.bindPositions,
    plan.boneA,
    plan.boneB,
    plan.weightB,
    controlPositions,
  );

  expandPositions(
    controlPositions,
    plan.renderToControlVertex,
    streams.positions,
  );

  computeSequentialFlatNormals(
    streams.positions,
    streams.normals,
  );
}
```

如果最终每个三角形都是三个独立顶点，那么 Flat Normal 的计算甚至不需要读 Index：

```ts
function computeSequentialFlatNormals(
  positions: Float32Array,
  normals: Float32Array,
): void {
  for (let p = 0; p < positions.length; p += 9) {
    // 读取三个顶点，计算叉乘和 normalize
    // 将同一法线写三次
  }
}
```

这会直接删除每帧的：

* Patch 类型分支；
* Quad 拆分；
* Writer 顶点计数维护；
* Writer triangle 调用；
* Surface 范围重新建立；
* Index 相关控制流。

## Color 也不应该每帧全量刷新

Vanguard 的 Skin、Hair、Tunic、Pants 等语义基本不随动画变化。

所以：

```text
semanticIds + palette
→ 初始化时生成 Color
```

只有以下情况再更新：

* 换皮肤；
* 装备变更；
* 受击闪烁；
* 全局染色；
* 特殊状态。

也就是说，普通 idle 动画每帧应当只上传：

```text
Position + Normal
```

而不是：

```text
Position + Normal + Color
```

进一步可以把颜色系统改成：

```ts
interface SemanticPalette {
  readonly colors: ReadonlyArray<ReadonlyVec4>;
}

function bakeSemanticColors(
  semanticIds: Uint8Array,
  palette: SemanticPalette,
  output: Uint8Array,
): void;
```

---

# 四、蜘蛛应该“预编译采样计划”，而不是每帧运行 Tessellator

蜘蛛的问题和 Vanguard 不完全一样。

Vanguard 是控制笼映射；蜘蛛是参数曲面求值。因此不要强行共用同一种 Plan，但可以共用相同的运行时契约。

## 蜘蛛当前重复执行的静态工作

每帧调用：

```text
appendCubicTube()
appendEllipsoid()
appendEllipsoid()
Triangle Fan
```

这里面有很多信息实际上永远不变：

* Tube 有几段；
* 每个截面有几个径向顶点；
* 截面之间如何连三角形；
* 椭球经纬采样的拓扑；
* 液体 Fan 的 Index；
* 各顶点的参数坐标；
* 径向角度的 sin/cos；
* 椭球采样角度的 sin/cos；
* 每个顶点属于 Body、Eye 还是 Liquid。

这些应该在初始化时编译。

## 推荐的数据结构

```ts
interface TubeSamplePlan {
  readonly vertexOffset: number;
  readonly segmentCount: number;
  readonly radialCount: number;

  // 每个截面的曲线参数 t
  readonly curveT: Float32Array;

  // 径向单位圆，初始化时计算
  readonly radialCos: Float32Array;
  readonly radialSin: Float32Array;
}

interface EllipsoidSamplePlan {
  readonly vertexOffset: number;

  readonly localDirections: Float32Array;
  readonly normalDirections: Float32Array;
}

interface CurveCrawlerMeshPlan extends MeshPlan {
  readonly legPlans: readonly TubeSamplePlan[];
  readonly abdomenPlan: EllipsoidSamplePlan;
  readonly thoraxPlan: EllipsoidSamplePlan;
  readonly eyePlans: readonly EllipsoidSamplePlan[];
  readonly liquidPlan: FanSamplePlan;
}
```

初始化时：

```text
CurveCrawler Recipe
→ compileTubePlan()
→ compileEllipsoidPlan()
→ compileFanPlan()
→ 合并固定 Index
→ CurveCrawlerMeshPlan
```

每帧：

```text
更新步态参数
→ 计算 8 条腿的控制点
→ evaluateTubePositionsAndNormals()
→ evaluateEllipsoidPositionsAndNormals()
→ evaluateLiquid()
→ 按 Dirty Mask 更新颜色
→ 上传变化流
```

## 不建议一开始做通用“几何虚拟机”

例如不要马上设计这种东西：

```ts
[
  OP_TUBE,
  OP_ELLIPSOID,
  OP_FAN,
  OP_TRANSLATE,
  OP_ROTATE,
  ...
]
```

除非未来会有大量怪物都由这些体元组合，否则它可能只是把函数调用变成 Opcode 分支，代码并不会更少。

更合适的是少量组合式 Kernel：

```ts
evaluateCubicTube(...)
evaluateEllipsoid(...)
evaluateTriangleFan(...)
```

Plan 数据统一，算法保持独立。

# 五、大厅不需要和动态角色完全统一

大厅目前的抽象其实是三者中最健康的：

```text
SurfaceFrame + Grid Recipe + Deformer
```

它已经把以下内容分开：

* 曲面的局部坐标系；
* 拓扑密度；
* 形变函数；
* 世界空间映射；
* 绕序和对角线策略。

观察窗后墙保留专门的径向拓扑也是正确的。

不要为了“调用树统一”而把它强行塞进 Grid 抽象。领域特例不等于架构失败。

大厅只需要接入新的编译接口：

```ts
const plan = compileLobbyRecipe(recipe);
const streams = evaluateStaticMesh(plan, recipe);
const mesh = createStaticMesh(plan, streams);
```

然后可以释放编译期间的临时采样缓存。

也就是说，三套系统统一的是：

```text
MeshPlan
VertexStreams
语义数据
GPU Adapter
```

而不是统一：

```text
所有形状的生成算法
```

# 六、推荐的新调用树

## 初始化阶段

```text
Domain Recipe / Cage
└─ Mesh Compiler
   ├─ 展开 Patch 或 Primitive 拓扑
   ├─ 生成固定 Index
   ├─ 生成顶点映射或参数采样计划
   ├─ 生成 Semantic ID
   └─ 生成 MeshPlan
      ↓
   Allocate VertexStreams
      ↓
   Initial Evaluator Pass
      ↓
   GPU Mesh Initialize
```

## 每帧阶段

```text
Population Systems
└─ 改变领域状态并设置 Dirty Flags
   ↓
Mesh Evaluator
├─ Geometry Dirty
│  ├─ 更新 Position
│  ├─ 更新 Normal
│  └─ 更新 Bounds
└─ Color Dirty
   └─ 更新 Color
      ↓
GPU Stream Uploader
└─ 只上传发生变化的 Attribute
```

Vanguard 最终可以缩到：

```text
VanguardPopulation.update()
├─ animation.update()
└─ mesh.update(Geometry)
   ├─ skinControlVertices()
   ├─ expandFlatMesh()
   ├─ Core writeSequentialFlatNormals()
   └─ upload(Position | Normal)
```

蜘蛛可以缩到：

```text
CurveCrawlerPopulation.update()
├─ behavior / movement / animation systems
└─ mesh.update(dirty)
   ├─ evaluateCrawlerSections()
   ├─ updateColorsIfNeeded()
   └─ upload(dirty)
```

调用树不只是视觉上缩短了，更重要的是每一层都对应明确的数据生命周期。

# 七、顶点模块建议这样分目录

不建议为每个概念都创建一个 class。用 TypedArray 数据结构和纯函数更适合你的项目。

```text
assets/core/mesh/
├─ mesh-plan.ts
├─ vertex-streams.ts
├─ mesh-dirty.ts
├─ mesh-runtime.ts
├─ mesh-assembler.ts
├─ semantic-colors.ts
└─ flat-mesh/
   ├─ flat-mesh-plan.ts
   ├─ flat-mesh-compiler.ts
   └─ flat-mesh-evaluator.ts
```

Cocos 相关内容留在：

```text
assets/core/rendering/cocos/
├─ cocos-static-mesh.ts
├─ cocos-dynamic-mesh.ts
└─ cocos-stream-uploader.ts
```

领域模块则变成：

```text
assets/player/vanguard/geometry/
├─ vanguard-cage.ts
├─ vanguard-mesh-plan.ts
├─ vanguard-mesh-compiler.ts
```

蜘蛛：


这里 `TriangleMeshWriter` 可以保留，但建议改名为：


# 八、几个具体的代码量削减点

## 1. 取消运行期 Patch 分支

把：

```ts
switch (patch.kind) {
  case Triangle:
  case Quad:
  case FacetedQuad:
}
```

移到编译阶段。

---

## 2. 用映射数组代替反复调用 Writer

把：

```ts
writer.vertex(...)
writer.vertex(...)
writer.vertex(...)
writer.triangle(...)
```

变成简单的紧凑循环：

```ts
for (let rv = 0; rv < renderVertexCount; rv++) {
  const cv = renderToControlVertex[rv];
  copyVec3(controlPositions, cv, positions, rv);
}
```

---

## 3. Semantic ID 代替 Section Range 控制流

相比：

```text
Skin range
Hair range
Tunic range
Scarf range
...
```

更通用的数据是：

```ts
semanticIds[vertexIndex] = VanguardSemantic.Hair;
```

Range 可以作为编译优化结果保留，但不应该成为语义的唯一表达。

如果同类顶点天然连续，可以同时输出：

```ts
interface SemanticSpan {
  semantic: number;
  vertexStart: number;
  vertexCount: number;
}
```

于是既有清晰语义，也能快速批量填色。

---

## 4. Static、Pose、Event 三种更新频率

建议不要只分 Static/Dynamic，而是至少分三档：

```ts
type UpdateFrequency =
  | "static"
  | "pose"
  | "event";
```

例如：

| 属性       | Vanguard | 蜘蛛     | 大厅     |
| -------- | -------- | ------ | ------ |
| Index    | static   | static | static |
| Position | pose     | pose   | static |
| Normal   | pose     | pose   | static |
| Color    | event    | event  | static |
| Semantic | static   | static | static |

这比笼统的 `Dynamic Mesh` 更能指导代码和上传策略。

---

## 5. 检查 Float64 是否真的必要

文档中 Vanguard 的变形共享点使用了预分配 `Float64Array`，但最终 GPU Position 基本还是 Float32 格式。

如果场景尺度正常，并且没有高精度世界坐标需求，可以测试直接使用 `Float32Array`：

* 少一份精度转换；
* 缓存占用更小；
* 与最终 GPU 数据一致；
* SIMD/批处理通常更自然。

这不是必须改，但很值得基准测试。不要仅因为 JavaScript Number 是双精度，就默认中间缓存也必须是 Float64。

---

# 九、不要过度抽象的地方

## 不要统一所有形状算法

Vanguard 的 Cage、墙壁的 Grid、蜘蛛的 Tube 本质不同。统一它们只会产生巨大的万能接口。

应该统一输出协议：

```text
Plan + Streams + Evaluator
```

而不是统一领域建模语言。

## 不要创建 `Vertex` 对象

避免：

```ts
class Vertex {
  position: Vec3;
  normal: Vec3;
  color: Color;
}
```

你现在使用 TypedArray/SoA 是正确方向。模块化不等于对象化。

## 不要让 Renderer 理解领域拓扑

Renderer 不应该知道：

* 什么是骨骼；
* 什么是 Bézier Tube；
* 什么是墙壁 Recipe；
* 什么是 Hair、Tunic、Liquid。

Renderer 只应该知道：

```text
有哪些流发生变化
应该上传哪些区间
使用哪个材质
```

## 不要为了消除特例而消除特例

观察窗后墙的径向拓扑应该继续是专用 Compiler。强行通用化通常会制造更多参数和分支。

---

# 十、最稳妥的迁移顺序

## 第一阶段：先拆生命周期，不改算法

新增：

```text
MeshPlan
VertexStreams
MeshDirty
```

让 Renderer 支持：

```ts
upload(streams, MeshDirty.Position | MeshDirty.Normal);
```

先停止无条件更新 Color。

## 第二阶段：编译 Vanguard Flat Topology

引入：

```text
renderToControlVertex
faceSemanticIds
固定 indices
```

把 `appendPatch()` 从每帧路径移走。

这是风险最低、收益最直观的一步。

## 第三阶段：把语义颜色变成 Event Stream

普通动画不再调用完整 `VertexShading.update()`。

## 第四阶段：编译蜘蛛 Primitive Plan

先做：

* 固定 Index；
* 参数坐标；
* sin/cos 表；
* vertex offsets。

保持原有 Bézier 和步态算法不变。

## 第五阶段：再重命名和精简 Renderer

当数据流已经分开后，再处理：

```text
GeometrySource
FixedTopologyBatchRenderer
DynamicMeshBatch
VertexShading
```

否则很容易只做一次“名称重构”，底层运行路径完全没变化。

---

# 最终判断

你当前架构不是“模块太少”，而是存在一个关键的中间表示缺失：

> **领域模型和最终 TypedArray 之间缺少 Compiled Mesh Plan。**

补上这一层后：

* `TriangleMeshWriter` 退回初始化阶段；
* 固定拓扑不再每帧重新遍历；
* Flat Shading 展开变成数组映射；
* 颜色从帧更新变成事件更新；
* 蜘蛛的 Tube/椭球变成预编译采样计划；
* Renderer 只管理 Stream 和上传；
* 大厅、玩家、蜘蛛可以共享运行时框架，同时保留各自最合适的建模算法。

最小而有效的核心抽象可以压缩成一句：

```text
Recipe --compile once--> MeshPlan
State  --evaluate-----> VertexStreams
Dirty  --upload-------> GPU
```

这会比继续增强 `GeometrySource.write(writer)` 更模块化，也更能真正减少动态路径的代码量和 CPU 工作。
