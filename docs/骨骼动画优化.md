## 总体建议

**应该重构，但不必立刻放弃程序化动画。**

最适合这个项目的是：

> **标准层级 Rig + 局部空间 Pose + 程序化/动画片段混合 + GPU 蒙皮。**

把“骨架如何传递变换”“姿态如何生成”“网格如何蒙皮”拆成三个独立模块：

1. **Rig 层**：标准父子层级、Bind Pose、局部 SRT、关节限制。
2. **Pose 层**：跑步、瞄准、武器动作、后坐、IK、惯性化。
3. **Rendering 层**：优先 `SkinnedMeshRenderer` GPU 蒙皮；必要时自定义 GPU Skinning。

不要继续让每块身体独立写世界矩阵。那种结构可以制造“动作”，但很难制造“重量”。

# 1. 是否应重构为标准层级人形 Rig？

**是。优先级最高。**

这里的“标准”不一定要求兼容某个 Humanoid Retargeting 标准，而是至少满足：

* 每根骨骼有明确父骨骼。
* 动画主要输出局部位置和局部旋转。
* 世界矩阵通过 FK 统一展开。
* 有固定 Bind Pose 和 inverse-bind matrix。
* IK、Additive、Mask 都针对同一套骨架定义。

推荐层级：

```text
CharacterRoot                 // 游戏逻辑、碰撞、导航的权威 Transform
├─ VisualRoot                 // 视觉转向、倾斜、受击偏移
│  └─ Pelvis
│     ├─ SpineLower
│     │  └─ Chest
│     │     ├─ Neck
│     │     │  └─ Head
│     │     ├─ Clavicle_L
│     │     │  └─ UpperArm_L
│     │     │     └─ LowerArm_L
│     │     │        └─ Hand_L
│     │     └─ Clavicle_R
│     │        └─ UpperArm_R
│     │           └─ LowerArm_R
│     │              └─ Hand_R
│     ├─ Thigh_L
│     │  └─ Calf_L
│     │     └─ Foot_L
│     │        └─ Toe_L
│     └─ Thigh_R
│        └─ Calf_R
│           └─ Foot_R
│              └─ Toe_R
├─ IKTargets
│  ├─ HandGrip_L
│  ├─ HandGrip_R
│  ├─ ElbowPole_L
│  ├─ ElbowPole_R
│  ├─ FootLock_L
│  ├─ FootLock_R
│  ├─ KneePole_L
│  └─ KneePole_R
└─ WeaponAimRoot
   └─ Weapon
      ├─ MainGrip
      ├─ SupportGrip
      ├─ Muzzle
      ├─ PumpHandle
      └─ MagazineSocket
```

关键是将 IK Target 放在被求解链之外，避免“手跟武器、武器又跟手”的循环依赖。

内部数学应变成：

```ts
worldPose[i] = worldPose[parent[i]] * localPose[i];
skinMatrix[i] = worldPose[i] * inverseBindPose[i];
```

程序化曲线也应输出 `localPose`，而不是直接输出世界矩阵。

这样做以后：

* 骨盆移动自然带动脊柱和腿根。
* 胸腔转动自然带动锁骨和手臂。
* 肩胯反向扭转可以连续传递。
* IK 能稳定地作用于完整链。
* 惯性化可以处理关节局部速度，而不是一组无关世界矩阵。

# 2. CPU 蒙皮还是接入 Skeleton/SkeletalAnimation？

## 推荐选择

**骨架接入 `Skeleton + SkinnedMeshRenderer`，姿态仍然可以完全由 TypeScript 生成。**

Cocos 3.8 的实时骨骼路径会把姿态写入场景骨骼节点，并从骨骼节点树提取矩阵上传 GPU；官方明确将“程序控制骨骼、IK、混合”等列为实时计算模式的用途。预烘焙模式则牺牲运行时可编辑性。 ([Cocos Creator][1])

因此：

* **渲染端使用 Skeleton/SkinnedMeshRenderer。**
* **动画源不必立即变成导入动画。**
* 仍可由 TS 每帧设置骨骼的局部位置、旋转。
* 若使用旧式 `SkeletalAnimation` 组件，需要关闭预烘焙：
  `useBakedAnimation = false`。
* 若改用 Marionette `AnimationController`，它不能与 `Animation` 或 `SkeletalAnimation` 挂在同一节点。 ([Cocos Creator][2])

`Skeleton` 资产本身保存骨骼相对 `skinningRoot` 的路径和 Bind Pose 矩阵；`SkinnedMeshRenderer` 负责骨架、蒙皮根节点和可蒙皮 Mesh 的组合。 ([Cocos Creator][3])

## 程序化 Mesh 的现实问题

Cocos 的 `MeshUtils.createMesh/createDynamicMesh` 支持运行时创建静态或动态 Mesh，但官方公开工作流主要面向 `MeshRenderer`，并不会自动替你生成 Skeleton、骨骼路径、inverse-bind 和蒙皮语义。 ([Cocos Creator][4])

所以建议按以下优先级实施：

### 最稳妥

在编辑器构建阶段生成角色 Mesh，然后导出或生成带有：

* joints
* weights
* inverse bind matrices
* bone hierarchy

的 glTF/FBX 资产，再由 Cocos 正常导入。

Mesh 几何仍然可以由你的算法生成，只是从“每次运行生成”改成“编辑器或构建期生成”。

### 必须运行时生成时

有两个选择：

1. 自己构建兼容 `SkinnedMeshRenderer` 的 Mesh/Skeleton 数据。
2. 保留自定义骨矩阵系统，但把蒙皮移到自定义顶点 Shader。

第二种通常比每帧 CPU：

* 遍历全部顶点；
* 蒙皮位置；
* 重算法线；
* 更新动态 Mesh；

更值得。

## Low Poly 硬分面法线

不要依赖每帧重新计算面法线来维持 Low Poly 外观。

更好的方案是：

* 每个面复制独立顶点，避免跨面共享法线。
* 在 Bind Pose 中保存每面固定法线。
* 法线使用和位置相同的骨权重进行 GPU 变换。
* 骨骼尽量只使用旋转和平移，不在动画中使用非均匀缩放。

如必须得到变形后三角形绝对精确的平面法线，可在片元 Shader 根据世界坐标导数计算几何法线，但需要针对目标平台验证性能和导数支持。

双骨骼权重完全够用；可写入四权重格式，但只让两个权重非零。

# 3. 原地动画还是 Root Motion？

对于顶视角 Run-and-Gun，**默认使用原地动画，游戏移动系统是权威来源。**

角色位置由：

* 输入；
* Character Controller；
* 碰撞；
* 导航；
* 网络同步；

决定。动画只去匹配实际速度，而不是反过来决定角色速度。

原因是这类游戏通常存在：

* 移动方向与瞄准方向不一致；
* 高频启动、急停、反向；
* 边跑边射击；
* 碰撞修正；
* 击退和外力；
* 多方向 strafing。

Root Motion 会让这些系统互相争夺角色位移权。

我在 3.8 官方 Animation Controller 文档中也没有找到完整的一等 Root Motion 提取/应用接口；公开控制接口主要是状态、变量、层权重和播放进度。因此在 3.8.8 上不宜把核心移动依赖在 Root Motion 工作流上。 ([Cocos Creator][2])

## Phase 不要只按时间推进

最好按实际移动距离推进：

```ts
phase += planarSpeed * dt / strideLength * Math.PI * 2;
```

更准确地说，使用：

```ts
distanceAccumulator += length(actualDisplacement);
phase = distanceAccumulator / strideLength * Math.PI * 2;
```

这样碰撞减速后脚步不会仍按输入速度狂奔。

还需要：

* 根据实际速度调节步长和抬脚高度。
* 播放速度只做有限缩放。
* 大速度范围使用 walk/run 两套形状混合。
* 前后左右使用方向参数，而不是简单旋转同一跑步姿态。
* 发生急转向时进行 phase 修正，优先保持当前支撑脚。

Root Motion 可只用于少量强设计动作，例如：

* 翻滚；
* 短距离突进；
* 处决；
* 近战冲刺。

即使这些动作也可以使用显式位移曲线，由游戏逻辑读取并执行，而不必依赖动画根骨直接接管移动。

# 4. 如何实现跑步、瞄准、双手 IK 和武器动作？

推荐把系统理解为一条 **Pose Pipeline**，而不是多个系统轮流覆盖骨骼。

## 每帧处理顺序

```text
1. 读取游戏状态和实际位移
2. 生成基础 locomotion pose
3. 混合启动、停止、转向和武器持枪基础姿态
4. 对状态切换做 inertialization
5. 处理支撑脚锁定、骨盆高度、腿部 IK
6. 分配上半身瞄准旋转
7. 计算武器目标姿态
8. 叠加后坐、摆动、泵动和换弹阶段
9. 解算双臂 IK
10. 应用关节限制、twist 分配和最终修正
11. 层级 FK，上传蒙皮矩阵
```

## 各层职责

| 层               | 内容                             |
| --------------- | ------------------------------ |
| Base Locomotion | 骨盆、腿、脊柱基础摆动、肩胯反向扭转、手臂自然摆动      |
| Weapon Hold     | 不同武器的胸腔、锁骨、肘部和手腕基础姿态           |
| Aim             | 将 Aim yaw/pitch 分配给骨盆、脊柱、胸腔和锁骨 |
| Weapon Action   | 泵动、换弹、拉栓、抛壳、武器下降               |
| Additive        | 后坐、呼吸、受击、微摆动、奔跑抖动              |
| Constraints     | 脚锁、地形 IK、双手武器 IK、关节限制          |
| Inertia         | 状态切换、急停、转身和武器重量造成的滞后           |

Cocos 3.8 的 Pose Graph 提供过滤混合、Animation Mask、Additive Blend 和双骨骼 IK，因此概念上可以表达这一流程。 ([Cocos Creator][5])

但 Pose Graph 在 3.8 中仍被标记为实验性；而且官方“动画图面板”页面称当前只支持一个动画图层，其他页面又展示层混合和 Additive Layer，文档存在明显版本表述差异。生产项目更稳妥的做法是：**核心组合在一张 Pose Graph 内完成，或继续由自己的 TypeScript Pose Pipeline 处理。** ([Cocos Creator][6])

## 武器应成为姿态权威

持枪状态下推荐：

> 武器目标姿态先确定，两只手再通过 IK 去抓武器。

而不是：

> 武器挂右手，左手追武器，右手又受瞄准系统修正。

前者更容易让枪口稳定指向目标。

### 普通持枪

1. `WeaponAimRoot` 根据瞄准方向确定旋转。
2. 添加武器类型的 shoulder offset。
3. 添加移动 sway、呼吸和后坐。
4. 由 `MainGrip` 和 `SupportGrip` 得到两个手部目标。
5. 分别使用两骨骼 IK 解算手臂。
6. 手腕最终对齐 Grip socket。

### 后坐

后坐最好不是直接旋转手骨，而是修改：

* WeaponAimRoot；
* 胸腔；
* 锁骨；
* 少量 VisualRoot。

然后再解算双手 IK。

这样枪和手不会脱离。

建议把后坐实现为冲量驱动的二阶弹簧：

```ts
recoilVelocity += shotImpulse;
recoilVelocity += (-stiffness * recoilOffset
                   - damping * recoilVelocity) * dt;
recoilOffset += recoilVelocity * dt;
```

枪口后坐可以快速，胸腔反应略慢，形成质量感。

### 泵动

* 武器主体仍由 WeaponAimRoot 控制。
* 左手目标跟随 `PumpHandle`。
* 动作曲线控制 PumpHandle 沿局部轴移动。
* 左臂 IK 自动跟随。
* 在泵动期间适当降低瞄准稳定性。

### 换弹

换弹不适合始终保持双手 IK 权重为 1。

使用阶段曲线：

```text
持枪
→ 左手 IK 权重下降
→ 左手去弹匣
→ 弹匣脱离
→ 新弹匣进入
→ 左手返回前握把
→ 左手 IK 权重恢复
```

右手可继续通过 IK 固定主握把；左手在换弹中间阶段由程序化轨迹或短动画片段驱动。

“动画驱动”和“IK 驱动”之间必须有连续权重，不要在某帧直接切换所有权。

# 5. 脚锁、IK、惯性化和肩胯联动放在哪里？

## 肩胯联动：基础 Locomotion 层

它属于跑步动作本身，不应作为最后的修饰。

最低关系可以是：

```ts
pelvisYaw = hipAmplitude * Math.sin(phase);
chestYaw  = -pelvisYaw * chestCounterRotation;
```

还应加入：

* 支撑腿侧骨盆轻微上提。
* 胸腔相对骨盆延迟少量 phase。
* 同侧锁骨不要完全跟随胸腔。
* 手臂摆动与对侧腿关联。
* 加速度产生前后倾。
* 横向加速度产生侧倾。
* 急停时胸腔和武器继续前冲后回弹。

这些关系共同构成“力量从地面经骨盆传到上身”的视觉。

## 脚掌锁定：基础姿态之后、腿 IK 之前

每只脚维护状态：

```ts
interface FootContactState {
    contactWeight: number;
    locked: boolean;
    worldLockPosition: Vec3;
    worldLockRotation: Quat;
}
```

进入支撑期时：

1. 记录当前脚掌世界位置。
2. 射线查询地面。
3. 保存 lock position 和地面法线。
4. 支撑期内维持该世界目标。
5. 离地前平滑降低权重。

顺序应是：

```text
确定两脚目标
→ 计算骨盆允许的高度范围
→ 调整骨盆
→ 解算大腿/小腿两骨骼 IK
→ 修正脚踝朝向
→ 最后处理脚尖
```

不要单独把脚拉到地面而不调整骨盆，否则膝盖容易过伸或跳动。

Cocos 3.8 的实验 Pose Graph 内置了带 Pole Target 的 two-bone IK，但生产中也可以继续使用自己的解析两骨 IK。 ([Cocos Creator][7])

## Inertialization：混合之后、IK 之前

惯性化主要解决：

* idle → run；
* run → stop；
* 前跑 → 后跑；
* 普通持枪 → 瞄准；
* 开火 → 恢复；
* 换弹阶段切换。

推荐在局部 Pose 上保存：

* 局部位置差和速度；
* 四元数差；
* 局部角速度。

旋转应在 quaternion log/exp 或 rotation-vector 空间衰减，不要直接对欧拉角逐轴平滑。

重要例外：

* 不要对已经锁定的脚掌世界位置做普通平滑，否则必然滑脚。
* 平滑 contact weight、骨盆修正量和锁定建立过程。
* 手部抓握目标同理：稳定握持时目标应固定，平滑权重而不是让目标不断追赶。

## 瞄准：脚和骨盆稳定后，上半身约束阶段

Aim yaw 不应全部压在胸骨上。按比例分配，例如：

```text
VisualRoot / Pelvis：处理大角度身体转向
SpineLower：15%
Chest：40%
Clavicles：15%
Neck/Head：剩余视觉跟随
```

当瞄准方向与下半身方向差值超过阈值，例如 60°～90°，应启动全身转向或踏步转向，不能无限扭腰。

# 6. 纯程序化动画的最低改造集

若暂时不引入任何动画片段，至少增加以下内容。

## 必需骨骼

* `VisualRoot`
* `Pelvis`
* 至少两段脊柱：`SpineLower + Chest`
* 左右锁骨
* 上臂、前臂、手
* 大腿、小腿、脚
* 脚尖
* 颈部和头部
* 独立 WeaponAimRoot
* 双手、双肘、双脚、双膝目标

Twist bone 不是第一阶段必需，但后续建议给前臂增加 twist 分配，否则双手握长枪时手腕非常容易翻折。

## 必需约束

1. 父子层级和局部空间 FK。
2. 肩、肘、髋、膝、腕的关节角限制。
3. 手臂和腿的 two-bone IK。
4. 稳定的 elbow/knee pole vector。
5. 支撑脚世界锁定。
6. 地面法线对齐和骨盆高度补偿。
7. Aim rotation 的分段分配。
8. 双手武器 Grip 约束。
9. 状态切换惯性化。
10. 加速度驱动的躯干倾斜和武器滞后。

## 最能消除机器人感的五项

按视觉收益排序：

1. **支撑脚真正锁在世界空间。**
2. **骨盆随支撑腿进行横向和高度转移。**
3. **胸腔和骨盆反向扭转，并有轻微相位延迟。**
4. **运动加速度影响躯干和武器，而不只影响腿部曲线。**
5. **两手通过 IK 服从一个有重量和后坐的武器目标。**

单纯增加更多正弦曲线通常不会解决问题；关键是引入：

* 接触；
* 约束；
* 层级传播；
* 时间滞后；
* 权威目标。

# 推荐迁移顺序

不要一次重写全部系统。

### 第一阶段：只重构姿态数学

保留当前 CPU 蒙皮，但改成：

```text
程序化曲线
→ 局部 Pose
→ 层级 FK
→ 当前 CPU Skinning
```

先验证骨盆、胸腔、四肢的力量传递。

### 第二阶段：加入脚锁和武器 IK

建立：

* contact state；
* pelvis solver；
* leg IK；
* WeaponAimRoot；
* 双臂 IK；
* recoil spring。

这是视觉提升最大的阶段。

### 第三阶段：迁移 GPU 蒙皮

把 Mesh 生成改到编辑器/构建期，接入 `Skeleton + SkinnedMeshRenderer`；或实现自己的 GPU skin shader，移除每帧 CPU 顶点和法线更新。Cocos 的实时骨骼路径就是为运行时骨骼修改和 GPU 蒙皮设计的。 ([Cocos Creator][1])

### 第四阶段：逐步引入短动画资产

不必先制作完整跑步动画库。可以优先引入最难纯程序化表现的部分：

* 换弹手部轨迹；
* 拉栓；
* 枪械检查；
* 受击；
* 死亡；
* 翻滚。

Locomotion、瞄准、脚部接触和后坐仍可保持程序化。

**最终架构应是 Hybrid，而不是“纯动画片段”与“纯程序化”二选一。**
