# 大厅帧率波动分析与优化策略

## 结论

大厅当前的低帧率不是由三角形数量或游戏逻辑主导，而是由以下三类问题叠加造成：

1. Cocos Creator 3.8.8 Web Pacer 在目标 60 FPS 与 60Hz rAF 同频时存在边界漏帧；
2. Web 首屏直接使用较高物理分辨率和 1024 Soft 2X 实时阴影，低端 GPU 容易错过显示窗口；
3. 原自适应分辨率会重复尝试已经失败的更高档位，在稳定档与掉帧档之间周期性震荡。

## 运行态证据

本地预览在约 1280×673 画布下复现了用户截图中的特征：

| 指标 | 观测值 | 判断 |
| --- | ---: | --- |
| Draw Call | 13 | 不属于提交批次数量瓶颈 |
| Triangle | 7418 | 不属于几何吞吐瓶颈 |
| Game Logic | 约 0.2 到 0.4ms | 逻辑和动态网格求值不是首要瓶颈 |
| Renderer | 约 0.6 到 1.5ms | CPU 渲染提交很轻 |
| Frame time | 约 0.8 到 2ms | 不能解释 20 到 50 FPS |
| Framerate | 约 38 到 56 FPS | 大量时间消耗在 Profiler 主动计时区间之外 |

Cocos Profiler 的 Frame time 从 `BEFORE_UPDATE` 统计到 `AFTER_DRAW`，不包含两帧之间的 Pacer、rAF、VSync 等待，也不能完整代表异步 GPU 压力。因此 `1000 / Frame time` 不能用于推断应有 FPS。

## Web Pacer 同频漏帧

Cocos Creator 3.8.8 的 `pal/pacer/pacer-web.ts` 使用累计时间计算理论帧编号：

```text
elapsedFrame = floor(elapsedTime / frameTime)
nextFrame = elapsedFrame + 1
```

目标 60 FPS 时 `frameTime` 约为 16.6667ms。60Hz rAF 也位于同一边界附近，微小时间抖动会让回调偶尔落到前一个理论帧编号，从而被跳过。

使用与引擎相同算法、60Hz 时间戳和 ±0.35ms 确定性抖动进行十秒隔离模拟：

- 调度目标 60：约 46 次游戏回调/秒；
- 调度目标 61：约 60 次游戏回调/秒。

因此 Web 调度目标改为 61。60Hz 设备仍由 VSync 限制在约 60 FPS，高刷新率设备则由 Pacer 控制在约 61 FPS。

## 像素与阴影成本

Web 画布使用物理像素，设备 DPR 会直接放大着色面积。原策略从 `shadingScale = 1` 启动，高 DPR 或高分辨率设备需要先经历数轮掉帧才会降档。

新策略：

- Web 从 0.85 启动，最低可降到 0.5；
- 根据物理画布尺寸和 120 万首屏像素预算进一步限制启动比例；
- 低于 40 FPS 时每次降低 0.2，快速退出严重掉帧区；
- 实时阴影默认关闭，避免 Shadow Caster Pass 持续占用低端 GPU；
- 调试面板开启阴影时，Web 使用 512 Hard，原生平台使用 1024 Soft 2X。

## 消除动态分辨率震荡

原策略在连续健康窗口后固定升档。如果升档后掉帧，会再次降档；恢复计数满足后又会重复尝试同一失败档位。这会造成：

- 周期性 FPS 高低切换；
- 反复修改 `pipeline.shadingScale`；
- 渲染附件调整带来的额外尖峰；
- 画面清晰度反复变化。

新控制器将升档视为一次恢复探测：

1. 保存探测前的稳定比例；
2. 升档后跳过一个采样窗口，让附件与 GPU 状态稳定；
3. 若随后不能达到恢复阈值，立即回退；
4. 把回退档记为本场景会话的恢复上限，不再重复探测同一失败档。

这样会优先维持稳定帧率，而不是持续追逐短暂的完整分辨率。

## 保留的优化边界

当前动态角色与蜘蛛已经使用固定拓扑、TypedArray、按脏流上传和低 Draw Call 合批。运行态 Game Logic 低于 0.5ms，继续牺牲动画更新率或大幅削减模型只会带来有限收益，因此本轮没有把主要改动堆到几何与动画系统。

后续若在目标低端设备上仍不能稳定，应先采集真实 GPU 型号、物理画布尺寸、稳定后的 `shadingScale` 与关闭阴影前后的 FPS，再决定是否增加“阴影完全关闭”的极低质量档；不应继续盲目削减三角形。
