## 1. Venom Lobber 的脚确实几乎没有动画

动画系统虽然会根据移动速度推进 `gaitPhase`：

```ts
phase += deltaTime * (1.4 + speed * 0.32);
```

也会生成轻微的身体上下起伏。

但渲染器对腿部所做的事情只有：

```ts
localX += cos(phase) * speedRatio * 0.18;
localZ += max(0, sin(phase)) * speedRatio * 0.28;
```

也就是把一整条腿的全部顶点作为一个整体，稍微向前后和平面上方移动。没有髋关节、膝关节、踝关节旋转，也没有脚掌固定。

而腿模型本身长度达到约 5 个局部单位，移动幅度只有 `0.18～0.28`，所以视觉上几乎像完全没有动。

### 正式修改：六足三段式步态

不要继续增加当前平移幅度，而是给每条腿建立三段程序化骨架：

```text
Coxa：身体连接关节
Femur：上腿
Tibia：下腿
Foot：接地点
```

模型编译阶段为腿部顶点增加：

```ts
interface VenomLegVertexBinding {
  legId: number;
  segmentId: number;
  segmentWeight: number;
}
```

六条腿使用交替三足步态：

```text
Tripod A：
左前 + 右中 + 左后

Tripod B：
右前 + 左中 + 右后
```

每个周期：

```text
60%：支撑阶段，脚掌锁定世界位置
40%：摆动阶段，脚掌沿抛物线移动到新落点
```

每帧先计算脚掌目标，再用二维解析 IK 求髋、膝、踝角度：

```ts
const knee = Math.acos(
  clamp((upper * upper + lower * lower - distance * distance)
    / (2 * upper * lower), -1, 1),
);

const hip = Math.atan2(targetZ, targetForward)
  - Math.acos(
    clamp((upper * upper + distance * distance - lower * lower)
      / (2 * upper * distance), -1, 1),
  );
```

随后按腿段 Pivot 旋转顶点。

这样会得到：

* 腿真正弯曲
* 脚落地时不滑动
* 前进速度越快，步幅越大
* 停止移动时，脚稳定停住
* 身体可以根据支撑三足产生轻微 Roll 和 Pitch

Venom Lobber 数量远少于普通蜘蛛，六腿解析 IK 的 CPU 成本可控。不要为了这个怪物接入 Cocos SkeletalAnimation，继续保持 TypedArray 程序化骨架更符合项目架构。

## 2. 出生横着钻出来的原因也在当前实现里

出生阶段的动画系统实际上只驱动：

* 尾部收卷
* 毒囊脉冲
* Body Bob 清零

它没有真正控制身体各部位的出土轨迹，也没有推进正常腿部步态。

渲染器则对整个模型统一执行：

```ts
const spawnTilt = (1 - spawnEase) * -0.64;
const spawnLift = (spawnEase - 1) * 8.6;
```

然后所有顶点一起旋转、一起向上移动。

`-0.64` 弧度大约是 `-36.7°`。这相当于把完整怪物侧着埋在地下，再把它作为一个刚体斜着抬上来，所以会有“横着从土里钻出来”的感觉。

尽管代码对尾部顶点使用了不同的上升权重，但身体仍然没有真正卷曲或分段展开。

### 正式出生演出：尾刺牵引式破土

保持当前 `1.6 秒` 总时长。

#### 0.00–0.25 秒：地下预兆

* 地面出现一个小型不规则隆起
* 尾刺尖端首先刺破地面
* 毒囊在地下短促发光一次
* 身体仍完全不可见

#### 0.25–0.78 秒：尾部牵引

* 尾部按 Bone 顺序逐节展开
* 尾刺向上并向后勾住
* 身体根节点沿一条三次 Bézier 曲线上升
* 身体只保持约 `10°` 的前倾，不再整体横倒

```ts
root = cubicBezier(
  undergroundPosition,
  tailPullPosition,
  raisedPosition,
  standingPosition,
  progress,
);
```

#### 0.78–1.35 秒：六足落地

按照：

```text
后腿 → 中腿 → 前腿
```

三组依次展开，每组间隔 `0.09 秒`。

直接复用正常移动的腿部 IK，让每只脚从身体下方移动到最终落点，而不是简单地把腿宽度乘以一个 Scale。

#### 1.35–1.60 秒：身体稳定

* 身体向下压一次
* 毒囊脉冲一次
* 尾刺抬至战斗待机姿势
* 脚掌保持固定
* 完成后进入 `Alive`

必须删除目前应用到所有顶点的全局 `spawnTilt`。出场姿态应该由：

```text
身体 Root 轨迹
尾部 Bone
六条腿 IK
毒囊 Scale
```

四部分共同构成。

# 3. 当前子弹确实不是实体碰撞

现在开火时直接执行：

```ts
monsters.damageAlongSegment(this.hitQuery, this.hits);
```

这条线段从枪口一直延伸到最大射程，伤害立即结算。

随后生成的 Projectile 只用于显示：

```ts
/** 推进全部无碰撞曳光；伤害已经在开火帧由 Hitscan 完成。 */
```

它不再进行任何怪物碰撞。

这会造成非常明显的时间错位。以 M4A1 为例：

```text
速度：38
射程：25
飞完整射程时间：约 0.66 秒
```

但敌人在第 `0 秒` 已经受伤，曳光要约 `0.66 秒` 才走完。

所以玩家会看到：

```text
怪物先受击或死亡
→ 子弹视觉随后才飞过去
```

这就是最主要的违和感。

# 4. 撤掉 Hitscan，使用实体弹丸 CCD

这里不应该改回简单的“每帧检查当前位置是否重叠”，高速子弹会穿模。

正确方案是：

## SoA 实体弹丸 + Swept Sphere CCD

每颗子弹具有真实状态：

```ts
interface ProjectileSoA {
  active: Uint8Array;

  previousX: Float32Array;
  previousY: Float32Array;
  previousZ: Float32Array;

  x: Float32Array;
  y: Float32Array;
  z: Float32Array;

  velocityX: Float32Array;
  velocityY: Float32Array;
  velocityZ: Float32Array;

  radius: Float32Array;
  remainingEnergy: Float32Array;
  remainingRange: Float32Array;

  hitCount: Uint8Array;
}
```

开火时只 Spawn，不结算伤害：

```ts
projectiles.spawn(
  muzzlePosition,
  direction,
  projectileDefinition,
);
```

每帧：

```ts
endPosition = startPosition + velocity * deltaTime;
```

然后检测的不是射线，而是：

```text
一个有半径的球体
从旧位置连续扫掠到新位置
```

扫掠形成的是 Capsule。

这叫连续碰撞检测 CCD。内部依然会使用一段运动路径进行数学求交，但它与 Hitscan 完全不同：

```text
Hitscan：
开火时从枪口瞬间检测到最大射程

CCD：
子弹拥有真实位置和飞行时间，
每帧只检测这一帧实际经过的空间
```

敌人只会在子弹真正飞到它的位置时受伤。

## 5. 宽相位：Supercover DDA 网格遍历

你现在的共享 Crowd 已经可以提供空间索引，但线段查询方式仍然是遍历整个线段 AABB 覆盖的矩形网格：

```ts
for (cellY = minimumCellY; cellY <= maximumCellY; cellY++) {
  for (cellX = minimumCellX; cellX <= maximumCellX; cellX++) {
    // 查询桶
  }
}
```

斜向长路径会查询大量实际上没有经过的 Cell。

应改为 **Supercover DDA**：

```ts
while (cellX !== endCellX || cellY !== endCellY) {
  visit(cellX, cellY);

  if (nextBoundaryX < nextBoundaryY) {
    cellX += stepX;
    nextBoundaryX += deltaBoundaryX;
  } else {
    cellY += stepY;
    nextBoundaryY += deltaBoundaryY;
  }
}
```

再根据：

```text
projectileRadius + maximumMonsterRadius
```

访问路径周围一圈或两圈 Cell。

这样复杂度接近：

```text
子弹实际穿过的网格数量
```

而不是线段包围矩形面积。

## 6. 窄相位：Swept Sphere 对 OBB/Capsule

不要使用怪物三角网格碰撞。那会非常贵，而且 Low Poly 模型的腿会产生大量无意义的命中。

采用复合碰撞体：

### Curve Crawler

```text
Thorax OBB
Abdomen OBB
```

### Venom Lobber

```text
前部甲壳 OBB
腹部 OBB
毒囊 Capsule
```

腿不参与子弹碰撞，避免子弹经常擦到细腿后停止。

### 求最早碰撞时间 TOI

把子弹的运动路径变换到怪物 OBB 局部空间，再把 OBB 扩大子弹半径：

```ts
expandedHalfExtentX = halfExtentX + projectileRadius;
expandedHalfExtentY = halfExtentY + projectileRadius;
expandedHalfExtentZ = halfExtentZ + projectileRadius;
```

随后用 Slab Algorithm 求：

```text
tEnter
tExit
```

若：

```ts
tEnter <= tExit
&& tEnter >= 0
&& tEnter <= 1
```

则 `tEnter` 就是本帧的准确碰撞时刻。

对移动怪物使用相对运动：

```ts
relativeStart = projectileStart - monsterPreviousPosition;
relativeEnd = projectileEnd - monsterCurrentPosition;
```

这样即使怪物在这一帧横向移动，也不会因为只使用当前坐标而漏判。

## 7. 穿透也要发生在真实碰撞过程中

Projectile 保留穿透能量：

```ts
remainingEnergy[slot] = definition.penetrationEnergy;
```

单帧碰撞循环：

```ts
let remainingTime = deltaTime;
let impacts = 0;

while (remainingTime > EPSILON && impacts < MAX_IMPACTS_PER_FRAME) {
  const end = position + velocity * remainingTime;

  collectSweptCandidates(position, end, radius);
  const nearestHit = findEarliestTimeOfImpact(...);

  if (nearestHit === null) {
    position = end;
    break;
  }

  position = lerp(position, end, nearestHit.time);
  applyDamage(nearestHit.entity, calculateDamage(remainingEnergy));

  remainingEnergy -= nearestHit.penetrationCost;
  impacts++;

  if (remainingEnergy <= 0) {
    deactivateProjectile();
    break;
  }

  position += direction * CONTACT_EPSILON;
  remainingTime *= 1 - nearestHit.time;
}
```

这样一颗 AKM 子弹会：

```text
真实飞到第一只怪
→ 发生碰撞
→ 扣除穿透能量
→ 继续飞行
→ 真实碰到第二只怪
```

而不是在开枪帧同时伤害四只怪物。

每颗 Projectile 维护一个固定的小型命中历史，防止同一帧或相邻帧重复命中同一实体：

```ts
hitPopulationIds: Uint16Array;
hitEntityIds: Uint16Array;
```

不创建 `Set` 或对象。

# 8. World 调度也需要调整

你已经有专门的 `SpatialIndex` 阶段。

但当前 Weapon 和 Monster 都在 `Combat` 阶段，而且 Weapon 的顺序是 `0`，Monster 是 `10`。也就是说武器先执行，怪物移动和 Crowd 重建随后才执行。

实体弹丸应重排为：

```text
Simulation
├─ MonsterMovementSystem
├─ MonsterAnimationSystem
└─ ProjectileIntegrationSystem

SpatialIndex
└─ MonsterSpatialIndexRebuildSystem

Combat
├─ ProjectileContinuousCollisionSystem
├─ PlayerWeaponSpawnSystem
└─ MonsterAttackSystem

PostSimulation
├─ DamageResolutionSystem
└─ LifecycleTransitionSystem

RenderPreparation
├─ ProjectileRenderSystem
└─ MonsterRenderSystem
```

更准确的顺序是：

```text
本帧怪物先移动
→ 空间索引按最新位置重建
→ 子弹从旧位置扫掠到新位置
→ 求最早 TOI
→ 结算伤害和穿透
→ 上传视觉
```
