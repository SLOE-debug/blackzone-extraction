# 原始问题分析

我看了提交 `281927c1`。这次提交虽然接入了怪物网格脏区上传，但同时删除了怪物视锥筛选、远距细节和渲染错峰。

## 结论

你现在看到的现象，**大概率不是“世界坐标越大越慢”**，而是两个问题叠加：

1. 怪物数量随游戏时间持续增加；
2. 所有驻留怪物每帧都被当成位置脏数据，完整计算并上传。

DC 一直稳定，是因为所有怪物仍被压在一个共享 MeshRenderer 中。**一个 Draw Call 可以绘制 20 只怪，也可以绘制 220 只怪**；DC 不变并不代表 CPU 顶点计算量、上传字节数和三角形数不变。

# 第一优先级：恢复视锥剔除，但不要恢复 LOD

你最新代码中的共享 Bounds 是：

```ts
minX: -1_000_000,
minY: -1_000_000,
maxX: 1_000_000,
maxY: 1_000_000,
```

所以这个怪物 Batch 基本永远处于可见状态。

而且当前共享渲染器已经明确从“可见怪物”改成了“所有具有可渲染生命周期的驻留怪物”。

这里要区分：

```text
视锥剔除：
屏幕外对象不计算、不上传、不绘制

LOD：
屏幕内远距离对象使用简化模型
```

你不想使用 LOD 没问题，但**视锥剔除必须保留**。

建议实现：

```text
Resident List
    ↓
Camera Frustum Visibility Bitset
    ↓
只对可见实体求值程序化网格
    ↓
Index Buffer 只包含可见实体
```

所有可见怪物仍使用完整模型，没有任何远距离简化。

### 更理想的实现

给每只怪物固定 GPU Slot：

```ts
const gpuSlot = populationBaseSlot + entityIndex;
const firstVertex = gpuSlot * verticesPerMonster;
```

可见性变化只修改 Index Buffer，不搬迁 Vertex Buffer。

否则继续使用紧凑布局时，一只怪物进入或离开可见集合，后面的怪物都会换位置，导致强制重写。

# 第二优先级：修复“整群 Dirty”

你现在的 Dirty Update 实际上还不是实体级 Dirty。

`CurveCrawlerPopulation.update()` 每帧都会调用：

```ts
this.rendering.update();
```

而 Render Handle 的 `update()` 只是：

```ts
this.owner.markDirty(this.entry);
```

这会把整个群体的单个布尔值设为脏。

随后 Dirty Plan 遍历所有 Resident；只要 `poseDirty` 为真，每个 Resident 都会变成 `Position` 更新：

```ts
this.updates[packedIndex] = poseDirty
  ? CurveCrawlerPackedMeshUpdate.Position
  : CurveCrawlerPackedMeshUpdate.None;
```

最终效果通常是：

```text
本帧整个群体 Dirty
→ 所有驻留怪物重新计算顶点
→ 所有连续位置数据合并成一个大范围
→ 上传整个驻留怪物位置流
```

共享渲染器确实会合并连续范围，但当所有实体都 Dirty 时，最后仍然是一个覆盖全部怪物的大范围。

所以自定义引擎的 offset 更新已经生效，但当前 Dirty 分布接近 100%，局部上传没有发挥真正价值。

## 应改成实体级 Dirty Bitset

例如：

```ts
poseDirty: Uint8Array;
colorDirty: Uint8Array;
topologyDirty: Uint8Array;
```

移动系统只标记实际移动的实体：

```ts
poseDirty[entityIndex] = 1;
```

动画系统只有在输出参数真正变化时标记：

```ts
if (
  Math.abs(nextPhase - oldPhase) > epsilon
  || Math.abs(nextBite - oldBite) > epsilon
) {
  poseDirty[entityIndex] = 1;
}
```

渲染完成后只清理实际处理的实体：

```ts
poseDirty[entityIndex] = 0;
```

不过需要注意：你的存活怪物基本都在移动和播放腿部动画，因此即使做成实体级 Dirty，很多怪物依然每帧都会脏。所以它必须和视锥剔除或固定渲染频率配合。

# 第三优先级：所有怪物统一使用 30 Hz 网格更新

这不是距离 LOD。

所有怪物保持相同质量，只把 CPU 程序化网格生成从每帧一次改为固定频率：

```text
游戏逻辑：60 Hz
怪物网格求值：30 Hz
渲染：60 Hz
```

或者目标本身是 30 FPS：

```text
怪物行为与分离：15–20 Hz
怪物移动：30 Hz
程序化网格：20–30 Hz
```

可以按页错开：

```ts
const shouldUpdatePage =
  (frameSequence + pageIndex) % updateInterval === 0;
```

例如 16 只怪物一页，不要一帧更新全部 14 页：

```text
Frame 0：Page 0、2、4、6……
Frame 1：Page 1、3、5、7……
```

这和原来的距离分档不同：

* 所有怪物使用完整模型
* 所有怪物使用相同更新频率
* 只是把 CPU 工作均匀分散到不同帧

# 为什么大约两分钟会升到 17 ms

普通怪物配置是：

```text
3 秒保护期
首批 16 只
每 4 秒增加 16 只
最大 220 只
```

波次公式会持续增加目标数量，直到达到容量。

普通怪物大约在：

```text
3 + 13 × 4 = 55 秒
```

达到 220 只。

Venom Lobber 则是：

```text
9 秒开始 2 只
每 11 秒增加 1 只
最大 18 只
```

因此在两分钟左右，场上目标数量大约已经达到：

```text
220 只普通怪
+
约 12 只 Venom Lobber
```

你当前又取消了视锥筛选，因此它们即使在镜头外，也仍可能参与模拟、完整模型求值和提交。

所以“走得越远”很可能只是“游戏运行得越久”。

# 第四优先级：环境完整窗口重建

这里可能是移动相关的第二个问题。

每次 Chunk 窗口变化，当前环境渲染器都会：

```ts
this.pendingBuilder =
  new BattlefieldEnvironmentWindowGeometryBuilder(...);
```

而 Builder 在构造时会：

1. 遍历整个活动窗口；
2. 统计全部顶点和索引；
3. 立即分配一份完整的新 Geometry；
4. 后续分帧写入所有环境实体。

完成后又创建一个新的 `DynamicMeshBatch`，再销毁旧 Batch。

当玩家持续行走时，这会不断产生：

```text
完整 Positions TypedArray
完整 Colors TypedArray
完整 Indices TypedArray
CPU Builder
新 Mesh
新 GPU Buffer
旧 Mesh 回收
```

如果在上一份 Builder 完成前又跨过 Chunk，旧 Builder 会直接被替换，其大块数组等待 GC。

这更容易造成：

* 周期性 FT 尖峰
* GC 后 FT 突然下降
* 连续移动比原地站立更差

## 推荐修改

使用持久化 3×3 或 5×5 Chunk Batch：

```text
Chunk Slot 0
Chunk Slot 1
...
Chunk Slot 24
```

跨区块时只更新进入窗口的一行或一列，不再重建整个窗口。

这一项应该排在怪物之后，因为它更可能造成移动期间的尖峰，而怪物增长更可能造成整体基线持续抬升。

## 实验 A：固定 16 只怪，持续行走两分钟

临时修改：

```ts
desiredPopulationCount = 16;
desiredVenomCount = 0;
```

结果：

* FT 不再持续增加：主要是怪物波次增长
* FT 仍增加：继续查环境重建或其他移动累积

## 实验 B：开局直接 220 只，原地站立两分钟

结果：

* 开局 FT 就很高，之后稳定：确定是怪物数量和完整网格

## 实验 C：关闭怪物，只持续跨 Chunk

结果：

* 出现周期性尖峰：环境完整窗口重建
* 平均 FT 逐渐升高且 GC 后下降：Builder 和 Mesh 分配压力

## 实验 D：220 只怪，但重新开启纯视锥剔除

保持完整模型，不启用距离 LOD。


# 建议新增的三个计数

你目前首先缺少的不是更多阶段计时，而是工作量计数：

```text
monsterEntitiesEvaluated
monsterPositionBytesUploaded
monsterPositionUploadCalls
```

再增加：

```text
environmentGeometryBytesAllocated
environmentBuilderReplacements
```

理想输出示例：

```text
Resident: 220
Frustum Visible: 74
Evaluated: 74
Position Upload: 1.8 MB/frame
Upload Calls: 5
```

这样一眼就能知道局部上传到底有没有生效。

# 推荐实际实施顺序

1. **固定怪物数量做 A/B 实验，确认时间增长和距离增长的区别。**
2. **恢复纯视锥剔除，不恢复距离 LOD。**
3. **把整群 `dirty: boolean` 改成实体 Dirty Bitset。**
4. **固定 GPU Slot，避免可见列表变化导致顶点搬迁。**
5. **按 16 只一页，把程序化网格统一降到 30 Hz 并错峰。**
6. **把环境改成持久化 Chunk Batch，消除整窗 Geometry 重建。**
7. **长期将腿部和身体程序化形变移动到 Vertex Shader。**
8. **Projectile 空间索引暂时不排第一；只有在持续射击时 FT 明显上涨，才优先处理。**

最值得立即改的不是自定义引擎本身，而是这一条链：

```text
所有驻留怪物
→ 每帧整群 Dirty
→ 完整程序化模型求值
→ 完整位置流上传
→ 单 Draw Call 绘制全部怪物
```

你的 offset Buffer API 已经把“怎么局部上传”解决了；现在真正需要解决的是**为什么每一帧几乎所有数据都被判定为需要上传**。
