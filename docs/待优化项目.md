# 一、宝箱按钮为什么还能控制右摇杆

## 当前设计背景

最新提交增加了 `VirtualJoystickGesture`，将同一个触摸分成两种结果：

```text
死区内按下并松开
→ 宝箱操作

拖出死区
→ 取消宝箱点击
→ 继续作为瞄准摇杆
```

这个行为不是偶发 Bug，而是提交中主动实现并写进测试的设计。状态机明确规定“一旦离开死区，本次手势永久转为摇杆拖动”。

测试也明确要求“拖出死区后继续保持触摸所有权”，也就是继续控制摇杆。

这与现在确认的产品规则冲突。

## 原始调用链

```text
BattlefieldSceneInteractionSystem.synchronize()
  └─ resolver.resolve(playerX, playerZ)
      └─ hud.setContextAction(OpenContainer)
          └─ aimJoystick.setActionIcon(OpenContainer)

Cocos TOUCH_START / TOUCH_MOVE
  └─ VirtualJoystick.handleTouchStart/handleTouchMove()
      ├─ updateFromTouch()
      │   └─ 写入 aimJoystick.value
      └─ gesture.move()
          └─ 拖出死区后取消 actionCandidate

BattlefieldControlHud.update()
  └─ writeAimState()
      └─ 读取 aimJoystick.value
          └─ state.aiming = true

BattlefieldControlWorldSystem.execute()
  └─ BattlefieldPlayerAimController.apply()
      └─ world.weaponFiringRequested = true
```

交互图标由 `BattlefieldSceneInteractionSystem` 写入 HUD。

问题的直接入口在 `VirtualJoystick.handleTouchStart()`：无论当前有没有宝箱图标，都会先调用 `updateFromTouch()`，因此宝箱模式下仍会写出摇杆轴值。

这些轴值随后被 `BattlefieldControlHud.writeAimState()` 转换成射击输入。

最后在 Input 阶段进入玩家瞄准和开火链路。

## 正确方案：显式互斥模式

不要继续使用“点击候选转拖动”的手势仲裁。右摇杆应有明确模式：

```ts
enum VirtualJoystickMode {
  Axis,
  Action,
}
```

行为规则：

```text
contextAction === null
→ Axis 模式
→ 可以瞄准、射击
→ 不产生宝箱操作

contextAction !== null
→ Action 模式
→ 轴值强制为 0
→ 不允许瞄准、射击
→ 松开时只产生一次宝箱操作
```

### 双重保护

不能只在 UI 触摸层处理，还需要在 HUD 层增加第二层保护：

```ts
private writeAimState(): void {
  if (this.contextAction !== null) {
    this.mutableState.aimX = 0;
    this.mutableState.aimY = 0;
    this.mutableState.aiming = false;
    return;
  }

  // 普通摇杆逻辑
}
```

这样即使以后 `VirtualJoystick` 再出现输入泄漏，只要操作图标存在，也不会产生射击。

## 需要修改的文件

| 文件                                                         | 修改                                                  |
| ---------------------------------------------------------- | --------------------------------------------------- |
| `assets/core/ui/virtual-joystick.ts`                       | 增加 Axis/Action 模式；Action 模式禁止调用 `updateFromTouch()` |
| `assets/core/ui/virtual-joystick-gesture.ts`               | 删除“拖出死区转摇杆”语义，或仅用于普通 Axis 模式                        |
| `assets/bundles/battlefield/ui/battlefield-control-hud.ts` | `contextAction !== null` 时强制清空瞄准输入                  |
| `tests/core/virtual-joystick-gesture.spec.ts`              | 把“拖出死区继续摇杆”测试改为“Action 模式始终不产生轴输入”                  |

# 二、子弹命中问题的背景调用链

最新提交已经把弹丸整体链路接通。当前正常调用链是：

```text
BattlefieldControlWorldSystem
  └─ BattlefieldPlayerAimController.apply()
      └─ world.weaponFireIntent

BattlefieldWeaponWorldSystem [PreSimulation]
  └─ BattlefieldPlayerWeaponRuntime.updateFiring()
      └─ BattlefieldWeaponAttackExecutor.execute()
          └─ BattlefieldProjectilePopulation.spawn()

BattlefieldProjectileIntegrationWorldSystem [Simulation]
  └─ BattlefieldProjectileCombatPopulation.integrate()
      └─ previousPosition → currentPosition

BattlefieldMonsterSpatialIndexWorldSystem [SpatialIndex]
  └─ Crowd 空间索引重建

BattlefieldProjectileCollisionWorldSystem [Combat]
  └─ BattlefieldProjectileCombatPopulation.collide()
      └─ BattlefieldMonsterTargetRegistry.findFirstProjectileHit()
          ├─ collectSegmentCandidates()       宽相位
          └─ group.writeProjectileHitForEntity()
              └─ CurveCrawlerProjectileHitSystem.findEntity()  窄相位

BattlefieldProjectileDamageWorldSystem [PostSimulation]
  └─ resolveImpacts()
      └─ BattlefieldMonsterTargetRegistry.damageMonster()
          └─ CurveCrawlerPopulation.damage()
```

武器现在已经被提前到 `PreSimulation`，新生成的子弹可以在同一帧进入积分。

弹丸积分发生在 `Simulation`。

碰撞发生在 `Combat`，并记录宽相位、窄相位和 Impact 数据。

伤害最后在 `PostSimulation` 路由。

所以现在的主要问题不再是“链路没调用”，而是 **窄相位命中体与屏幕视觉不一致**。

# 三、截图里的子弹为什么看起来应该命中，却没有命中

代码中存在两个明确问题。

## 问题 1：蜘蛛腿部被刻意排除在命中体之外

`CurveCrawlerProjectileHitSystem` 的文件注释明确写着：

> 使用不包含腿部的胸腔、腹部复合 OBB。

实际只计算了两个盒体：

* thorax 胸腔；
* abdomen 腹部。

截图中蜘蛛的可见面积很大一部分来自六条腿，但命中区域只覆盖中间的小块身体。玩家看到子弹掠过蜘蛛轮廓，会自然判断命中；窄相位却会返回 `false`。

更关键的是，测试当前主动固定了这个错误体验：

```text
“远离胸腔和腹部但穿过腿部时保持不命中”
```

另一个测试还要求碰撞体“不会扩张到腿部”，并验证一个稍远的弹道必须失败。

因此，最新提交虽然声称补充了完整伤害链路测试，但测试验证的是“正中心射击一定命中”，不是“玩家视觉上擦过怪物也应该命中”。完整链路测试里的弹丸全部沿怪物中心线飞行。

### 判断

你截图中标出的两发子弹很可能：

```text
进入了 Crowd 宽相位
→ 找到了蜘蛛候选
→ 但只擦过腿或身体外缘
→ CurveCrawlerProjectileHitSystem.findEntity() 返回 false
→ narrowPhaseHits 没有增加
```

---

## 问题 2：视觉子弹的前端领先于真实碰撞位置

当前可视子弹的局部前后长度为：

```text
最前端：+0.34
最后端：-0.27
```

也就是说，可视模型总长约 `0.61` 世界单位，并且有 `0.34` 单位显示在权威弹丸位置的前面。

渲染时，`state.x/y/z` 被当作模型原点，然后将这个向前 `0.34` 的尖端画出来。

但碰撞使用的是从 `previousX/Y/Z` 到 `x/y/z` 的弹丸中心扫掠，半径只来自武器的 `impactRadius`。

例如沙漠之鹰的碰撞半径只有 `0.12`。

因此会出现：

```text
屏幕上的黄色弹头已经进入蜘蛛
但碰撞中心还在蜘蛛前方约 0.34 单位
所以当前帧还没有命中
```

这也是截图中“明明重叠却没有伤害”的来源之一。

仅凭单张截图无法确定每一发分别属于“腿部未计算”还是“视觉弹头领先”，但当前代码中这两处都足以稳定制造该现象。

# 四、命中系统的修复方案

## 第一层：重新定义玩家可感知的 Curve Crawler 命中体

不要恢复成覆盖所有脚尖的巨大单盒体，那会让明显射偏的子弹也命中。

推荐使用三层复合体：

```text
胸腔 OBB
腹部 OBB
近端腿部宽容碰撞体
```

近端腿部碰撞体可以是：

* 一个较扁的旋转椭圆/OBB；
* 只覆盖身体到腿长约 55%～65% 的范围；
* 竖直高度保持较低，避免子弹从蜘蛛上方经过也被误判；
* 不需要覆盖最末端脚尖。

伪逻辑：

```ts
const bodyContact = min(thoraxContact, abdomenContact);

const forgivingFootprintContact = findSweptSphereBoxContact(
  ...,
  bodyLength * 0.55,
  bodyWidth * 0.45 + legLength * 0.55,
  bodyWidth * 0.35,
  impactRadius,
);

return min(bodyContact, forgivingFootprintContact);
```

更精确的版本可以为左右三组腿各放一个 Capsule，但考虑当前怪物数量和热路径，先使用一个扁平旋转 OBB 会更合适。

### 修改文件

`assets/bundles/common-monsters/entities/curve-crawler/population/curve-crawler-projectile-hit-system.ts`

当前只读取 `bodyLength`、`bodyWidth`。修复后需要同时读取：

```ts
legLength
legWidth
crouchAmount
biteAmount
turnAmount
```

并把可感知腿部范围加入碰撞体。

---

## 第二层：让视觉弹头不领先于碰撞点

权威 `state.x/y/z` 应代表 **弹头最前端**，而不是视觉模型的中心。

最直接的处理是把整个视觉弹体移到当前位置之后：

```text
当前：-0.27 ～ +0.34
修改：-0.61 ～ 0
```

也就是保证：

```text
visualMaximumForward <= 0
```

这样只要黄色弹头已经进入怪物，碰撞扫掠终点也至少已经到达同一位置。

修改位置：

`assets/bundles/battlefield/equipment/projectile/geometry/battlefield-projectile-geometry.ts`

可以在写顶点时统一减去头部偏移：

```ts
const collisionAlignedForward = localForward - 0.34;
```

不建议把碰撞线段向前扩大 `0.34`，因为那会把纯视觉长度变成额外射程，导致子弹提前伤害。应当移动视觉，而不是扩大物理。

---

## 第三层：保留并利用新加的链路统计

最新提交已经增加了以下数据：

```text
projectilesSpawned
projectilesIntegrated
broadPhaseCandidates
narrowPhaseHits
impactsQueued
damageEventsApplied
```

碰撞人口在每次候选、命中、Impact 和伤害路由时都会写统计。

针对截图场景，可以按这个表定位：

| 数据                                                | 判断                         |
| ------------------------------------------------- | -------------------------- |
| `broadPhaseCandidates = 0`                        | Crowd 空间索引或坐标转换错误          |
| `broadPhaseCandidates > 0`, `narrowPhaseHits = 0` | 蜘蛛碰撞体过小、腿部未覆盖或 Y 高度不交叉     |
| `narrowPhaseHits > 0`, `impactsQueued = 0`        | 穿透循环或命中历史错误                |
| `impactsQueued > 0`, `damageEventsApplied = 0`    | populationId/entityId 路由错误 |

从当前实现和截图表现判断，最高概率是第二行。
