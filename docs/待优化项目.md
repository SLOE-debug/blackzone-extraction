## 结论

**你的判断是对的。** 当前很多“子弹从怪物头顶飞过”的情况，核心不是 CCD 没检测到，而是上游没有成功选择纵向目标，随后代码主动进入了：

```text
targetElevation = null
→ targetY = muzzleY
→ directionY = 0
→ 子弹以枪口高度水平飞行
```

最新提交 `ba6be7f71e29c21e05261089679cca39d3713896` 修复了交互互斥、蜘蛛近端腿部命中体和弹头视觉对齐，但**没有重构纵向目标的查询逻辑**。

所以最新状态更准确地说是：

> 碰撞体已经变宽、视觉也对齐了，但大量子弹在进入碰撞检测前，Y 方向就已经算错了。

# 当前 origin call tree

## 1. 输入阶段：从玩家脚底查询目标

```text
BattlefieldControlWorldSystem.execute()
  └─ BattlefieldPlayerAimController.apply()
      └─ monsters.resolveAimTarget(
           player.positionX,
           player.positionZ,
           manualDirectionX,
           manualDirectionZ
         )
```

`player.positionX/Z` 表示的是主角**脚底位置**。

目标找到时：

```text
targetDistance =
  dot(monsterPosition - playerFootPosition, manualDirection)

fireIntent.targetElevation = monster.y
fireIntent.targetDistance = targetDistance
```

目标没找到时：

```text
fireIntent.targetElevation = null
fireIntent.targetDistance = null
```

## 2. 武器阶段：从真实枪口发射

```text
BattlefieldWeaponWorldSystem
  └─ player.writeWeaponRigPose()
      └─ writeBattlefieldWeaponMuzzlePose()
          └─ BattlefieldWeaponAttackExecutor.execute()
```

执行器的当前逻辑是：

```ts
const targetDistance =
  intent.targetDistance ?? weapon.projectile.maximumRange;

targetX = muzzleX + directionX * targetDistance;
targetY = intent.targetElevation ?? muzzleY;
targetZ = muzzleZ + directionZ * targetDistance;
```

也就是说，没有候选怪物时，目标 Y 确实直接等于枪口 Y，得到完全水平的弹道。

这不是偶发行为，现有测试还明确固定了“没有候选怪物时请求枪口高度的水平射击”。

---

# 为什么画面里明明有怪，系统却认为“没有目标”

## 根因一：纵向目标范围只有 16，M4A1 射程却是 25

当前纵向候选参数被固定为：

```text
角度：6°
距离：16
```

截图右上角是 `5.56×45`，对应当前 M4A1。M4A1 的实际弹丸射程为 25。

因此：

```text
0 ～ 16：可能获得怪物 Y 修正
16 ～ 25：必定没有 Y 修正，始终水平射击
```

M4A1 后 **9 个世界单位，也就是 36% 的有效射程**，当前必然以枪口高度平射。

你截图中红框所在的远端尸群，非常符合这个问题：子弹水平进入远处怪群，但 Curve Crawler 的身体明显低于人物持枪高度，所以会直接从上方掠过。

---

## 根因二：查询检测的是“怪物中心是否位于 6°锥形内”

当前目标查询不是检测“右摇杆射线是否经过怪物轮廓”，而是检测：

```text
怪物中心与瞄准方向的夹角是否小于 6°
```

`CurveCrawlerTargeting.findEntity()` 只拿怪物的中心点计算 alignment；即使手动射线穿过蜘蛛身体边缘或腿部宽容碰撞体，只要中心点不在 6°内，就返回 `false`。

这和最新提交增加的碰撞体发生了语义分裂：

```text
纵向目标查询：只看怪物中心点
实际弹丸碰撞：胸腔 + 腹部 + 近端腿部宽容 OBB
```

最新碰撞窄相位已经包含三个碰撞区域，并且是完整的三维检测。

于是会出现：

```text
XZ 平面上，子弹方向实际经过怪物宽容碰撞范围
→ 但怪物中心不在 6°锥内
→ 没有取得目标 Y
→ 子弹保持枪口高度
→ 三维 CCD 正确判断它从怪物上方飞过
```

这种情况下，CCD 返回“不命中”反而是正确的；错误发生在此前的纵向弹道求解。

---

## 根因三：目标距离以玩家脚底为 origin，但弹丸以枪口为 origin

这是另一个很严重、尤其影响近距离怪物的问题。

目标距离当前这样计算：

```text
player foot → monster
```

但弹丸实际从：

```text
muzzle → target
```

发射。

M4A1 的枪口相对 WeaponAimRoot 向前偏移 `0.978`。

假设怪物位于玩家正前方 4 单位：

```text
代码认为下降距离：4
枪口到怪物实际距离：约 4 - 0.978 = 3.022
```

代码却把怪物的 Y 放到枪口前方 4 单位的位置。结果是弹道下降得过慢；当子弹到达怪物实际所在的平面时，它还没有降到怪物高度。

粗略估算，如果枪口与怪物瞄准点高度相差 2 单位：

```text
高度误差 ≈ 2 × 0.978 / 4
        ≈ 0.49
```

半个世界单位的纵向误差，足够让子弹越过低矮蜘蛛的整个身体。

所以现在同时存在：

* **远距离**：超过 16 后完全没有 Y 修正。
* **近距离**：即使找到了目标，也可能因为 origin 不一致而下降不足。

---

## 根因四：选择的是“角度最佳目标”，不是射线最先经过的目标

当前 Registry 给候选目标打分：

```text
score = 角度误差 + 距离惩罚 × 0.08
```

角度优先级远大于距离。

在密集尸群中，可能发生：

```text
近处怪物稍微偏离中心
远处怪物中心更贴合右摇杆方向
→ 系统选择远处怪物的高度和距离
```

但射击水平方向始终不变。这会让子弹按远处目标的缓慢下降轨迹飞行，从近处低矮怪物上方经过。

纵向修正应该选择的是：

> 手动射线最先经过的怪物，而不是中心角度最漂亮的怪物。

---

# 正确修复方案

## 方案核心：把纵向目标解析移动到真实枪口之后

现在的结构是在 Input 阶段用玩家脚底提前算目标。应改为：

```text
Input
└─ BattlefieldPlayerAimController
   ├─ 只写手动 directionX / directionZ
   └─ 写 firingRequested

PreSimulation
├─ BattlefieldPlayerWorldSystem
│  └─ 更新玩家动画和 WeaponAimRoot
│
└─ BattlefieldWeaponWorldSystem
   ├─ 读取真实 WeaponAimRoot
   ├─ 计算真实 muzzleX/Y/Z
   ├─ 用枪口作为 origin 查询纵向目标
   └─ 生成弹丸
```

新的查询应该是：

```ts
resolveFireElevationAlongRay(
  muzzleX,
  muzzleZ,
  manualDirectionX,
  manualDirectionZ,
  weapon.projectile.maximumRange,
  result,
)
```

这样可以同时解决：

* 玩家脚底与枪口 origin 不一致；
* 每种武器射程不同；
* 近距离下降不足；
* 远距离超过 16 后永远水平射击。

---

## 目标查询必须从“角度锥”改为“射线走廊”

不要简单把 6°改成 10°或把 16改成25。那只能缓解，不能解决语义错误。

正确语义应该是：

```text
从真实枪口沿玩家手动 XZ 方向发出一条线段
→ 长度使用当前武器射程
→ 查询线段实际经过的怪物平面碰撞范围
→ 选择最早进入射线走廊的怪物
→ 只读取它的 Y
→ XZ 方向完全不变
```

项目已经有 `collectSegmentCandidates()`，现在弹丸宽相位就在使用它。

因此不需要新建另一套空间索引，可以复用：

```text
PlanarCrowdSeparationSystem.collectSegmentCandidates()
```

建议流程：

```text
muzzle XZ
→ muzzle XZ + manual direction × weapon range
→ Supercover DDA 查询候选
→ 各怪物执行平面 footprint 窄相位
→ 选择 segmentProgress 最小的目标
→ 返回 targetElevation
```

这通常还会比当前的圆形查询更省，因为当前代码会先收集玩家周围半径 16 内的全部怪物。

在截图这种高密度尸群中，圆形查询候选很多，而射线走廊只会关注玩家真正朝向的狭窄区域。

---

# FireIntent 数据结构也要调整

当前结构保存：

```ts
targetElevation
targetDistance // 从玩家脚底计算
```

建议改成：

```ts
interface BattlefieldFireIntent {
  directionX: number;
  directionZ: number;
  elevationTarget: {
    x: number;
    y: number;
    z: number;
  };
}
```

输入阶段只保存手动 `directionX / directionZ`；只有真实枪口射线找到目标后，武器阶段才构造完整 `BattlefieldFireIntent`。

注意，目标 XZ **不用于修改射击方向**，只用于在真实枪口阶段重新计算投影距离：

```ts
const distanceFromMuzzle =
  (target.x - muzzle.x) * directionX
  + (target.z - muzzle.z) * directionZ;

const targetX = muzzle.x + directionX * distanceFromMuzzle;
const targetZ = muzzle.z + directionZ * distanceFromMuzzle;
const targetY = target.y;
```

这样仍完全符合你的要求：

```text
玩家往哪拖右摇杆，就严格朝哪射击
怪物只贡献 Y
绝不吸附 XZ
```

---

# 没有目标时应该怎么办

**没有目标时不发射。**

输入阶段仍可记录玩家的持续开火请求，但武器阶段必须先从真实枪口解析到射线目标；解析失败时不构造完整射击意图、不生成弹丸，也不消耗弹药。

# 调试观测建议

临时给每次射击增加这些数据：

```text
elevationCandidateFound
elevationCandidatePopulationId
elevationCandidateEntityId
elevationForwardDistance
muzzleY
targetY
projectileDirectionY
firingSuppressedNoTarget
```

并在 Debug 视图显示：

```text
白线：手动 XZ 射线
小点：真实枪口 origin
横线：最终弹丸 Y 轨迹
盒体：被选中的怪物纵向目标
```

截图场景复现后，大概率会看到两种数据之一：

```text
firingSuppressedNoTarget = true
```

说明怪物被 16 距离、6°中心锥过滤掉；

或者：

```text
candidateFound = true
但 elevationForwardDistance 是从玩家脚底计算
```

说明近距离枪口 origin 偏差导致子弹还未下降到怪物高度。

## 优先级

1. **先把纵向查询 origin 改成真实枪口。**
2. **查询距离改成当前武器实际射程。**
3. **中心角度查询改成射线走廊与碰撞 footprint 相交。**
4. **选择射线最早经过的目标。**
5. 真正无目标时禁止生成弹丸和消耗弹药。

修完之后，碰撞调用链会变成：

```text
真实枪口
→ 手动 XZ 射线
→ 找到射线经过的第一只怪物
→ 只修正 Y
→ 生成正确三维弹道
→ Swept Sphere CCD
→ 命中并结算伤害
```

这次问题的主要责任点已经从 `CurveCrawlerProjectileHitSystem` 转移到了 **`BattlefieldPlayerAimController` 的目标查询时机和 origin**。当前 CCD 很可能只是正确地报告：“这颗水平子弹确实从怪物头顶飞过去了。”
