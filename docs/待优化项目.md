# 长期方案实现结果

当前工作区已经完成本文方案 C，不再把固定 Facet Shade 当作最终实现：

- [x] 共享战场批次改为“静态局部 Bind Pose + 每实体姿态纹理 + Vertex Shader 程序化形变”。
- [x] 每只 Curve Crawler 固定占用 `23` 个 `RGBA32F` Texel，保存位置、朝向、步态、出生、死亡碎块、受击和液化参数。
- [x] 稳定 GPU Slot 与姿态纹理行一一对应；视锥压缩只改变活动索引，不改变实体参数身份。
- [x] Body、Eye、Leg、Foot、Liquid、Crack、Egg 和 Shard 全部生命周期拓扑只在批次结构创建时生成局部 Bind Pose。
- [x] Vertex Shader 对 Position 和 Normal 执行同源姿态、碎块和实体朝向变换，不会再出现几何更新后保留历史法线的问题。
- [x] Fragment Shader 根据当前世界法线计算四档明暗；受击和液化颜色在 Shader 中由姿态参数混合，不再逐帧上传 Color。
- [x] CPU 逐帧不再求值完整蜘蛛顶点，也不再上传 Position/Normal；只整体上传约 `容量 × 23 × 16` 字节的姿态纹理和生命周期变化时的活动索引前缀。
- [x] 原固定分面着色器、60 Hz CPU 网格调度、Dirty Range 和共享 Unlit 顶点流实现已删除，避免保留两套最终方案。

基础灰紫配色仍保留，但最终浅暗关系来自当前 GPU Normal，而不是静态随机档位或某次 CPU 完整着色的历史状态。性能日志也已改为统计 GPU 姿态纹理上传量，不再报告已移除的 Position 局部上传。

## 同一轮保留的架构改造

- `CurveCrawlerPoseSnapshot` Hash 已删除；独立 Lit 展示渲染器仍用 Change Journal 观察低频颜色事件。
- 稳定槽位分配、无分配可见集合和轻量 `WorldPhase + WorldSystem + WorldScheduler + BattlefieldWorld` 继续保留。
- Scene Runtime 每帧只调用 `world.step(...)`，领域系统直接遍历 SoA TypedArray。
- 没有添加尚无真实调用点的对象式 Query 或 Command Buffer。

自动化测试已覆盖 GPU 姿态 Texel 布局、稳定 Slot 行、完整生命周期 Bind Pose 和变形元数据。仍需在 Cocos Creator 导入新 Effect 后，用原截图机位确认分档光照方向与蜘蛛动作幅度。

# 原始问题分析

是的，截图里同一种蜘蛛明显出现了**浅紫、深紫、接近黑色**的差异。红框里一只偏亮、一只几乎融进地面，这不只是审美感觉，原颜色更新代码确实存在逻辑不一致。

## 主要原因：几何姿态变了，但烘焙明暗没有跟着变

你的蜘蛛使用的是 CPU 计算法线，再将固定世界方向光照乘进顶点颜色：

```ts
const diffuse =
  normalX * LIGHT_DIRECTION_X
  + normalY * LIGHT_DIRECTION_Y
  + normalZ * LIGHT_DIRECTION_Z;

streams.colors[r] *= shade;
streams.colors[g] *= shade;
streams.colors[b] *= shade;
```

也就是说，最终颜色包含了基于法线计算的“假光照”。

但最新 Dirty 路径中：

```ts
Position 更新：
  writeGeometry = true
  writeColors = false
  writeNormals = false

Shaded 更新：
  writeGeometry = true
  writeColors = true
  writeNormals = true
```

普通姿态变化大部分只会成为 `Position`，不会重新计算法线和顶点色。

问题在于蜘蛛的朝向、腿部曲线、身体旋转都在改变世界空间几何。身体求值中明确使用了 `headingCosine`、`headingSine` 和碎片旋转来生成腿、脚和身体。

结果就是：

```text
蜘蛛转向或迈腿
→ Position 发生变化
→ 法线仍是上次 Shaded 时的法线
→ 顶点颜色仍是上次姿态的明暗
→ 不同蜘蛛停留在不同历史光照状态
```

这正好符合截图里的表现。

---

## 为什么 Color Dirty 没发现这个问题

`CurveCrawlerColorSnapshot` 目前只观察：

```text
hitFlash
liquidDrain
```

它并不观察：

* heading
* animation phase
* crouch
* bite
* turn
* fragment rotation
* 腿部姿态

所以渲染器认为：

```text
颜色事件没变
→ 不需要重新着色
```

但你的颜色不仅包含“固有色”，还包含了依赖姿态法线的世界方向光照。因此当前 Dirty 定义与颜色数据的真实依赖关系不一致。

# 正确性修复方案

把 Pose 变化进一步拆分：

```ts
enum CurveCrawlerGeometryChange {
  None = 0,
  Translation = 1 << 0,
  NormalAffectingPose = 1 << 1,
  BaseColor = 1 << 2,
  Topology = 1 << 3,
}
```

其中：

### 只影响位置、不影响法线

```text
transform.x
transform.y
```

纯平移不需要重新计算明暗。

### 会影响法线和明暗

```text
heading
phase
bodyPulse
crouch
bite
turn
fragmentRotation
fragmentScale
surfaceCollapse
emergence shape
```

Dirty Plan 应改成：

```ts
if (
  force
  || visibilityChanged
  || colorChanged
  || normalAffectingPoseChanged
) {
  update = CurveCrawlerPackedMeshUpdate.Shaded;
} else if (translationChanged) {
  update = CurveCrawlerPackedMeshUpdate.Position;
} else {
  update = CurveCrawlerPackedMeshUpdate.None;
}
```

这能修正画面，但对持续播放动画的蜘蛛来说，`phase` 每帧都变化，最终大部分蜘蛛仍然需要频繁 `Shaded`，性能代价会明显增加。

# 更适合你项目的修复：不要把世界光照烘进动态 Color

你的蜘蛛是程序化动态模型，而且你又希望所有怪物保持高帧率。CPU 顶点色不应该同时承担：

```text
固有颜色
+
受击颜色
+
世界方向光照
```

建议拆成：

```text
Color Buffer：
只存固有颜色、受击闪烁、死亡颜色

Lighting：
在 Shader 内根据 Normal 计算
```

## 方案 A：上传动态 Normal

动态顶点布局改成：

```text
Position
Normal
Color
```

姿态变化上传：

```text
Position + Normal
```

受击变化上传：

```text
Color
```

Shader 中计算：

```glsl
float diffuse = max(dot(normalize(v_normal), lightDirection), 0.0);
float shade = quantizeShade(diffuse);
finalColor = vertexColor * shade;
```

优点：

* 颜色不会停留在历史姿态
* Dirty 语义变得正确
* Hit Flash 不需要重新烘焙明暗
* 同类蜘蛛光照一致

缺点：

* 每次姿态更新需要多上传一份 Normal
* CPU 仍然需要计算动态法线
* 220 只怪物 60 FPS 时仍比较重

---

## 方案 B：固定分面明暗，不使用世界方向

对于低多边形 Unlit 风格，更简单的方式是让每个顶点保存一个稳定的静态明暗值：

```text
基础颜色 × 固定 Facet Shade
```

不要根据蜘蛛当前朝向重新计算世界方向光照。

例如：

```ts
const STATIC_SHADE_BY_SEMANTIC = [
  0.74,
  0.68,
  0.82,
  0.76,
];
```

这样蜘蛛转身时明暗不会跟着世界光变化，但会保持稳定的 Low Poly 分面感。

优点：

* Position 更新不需要 Color
* 不需要上传 Normal
* 没有颜色不同步
* 很适合大量怪物
* 视觉稳定

对你的项目而言，这是当前最务实的方案。

---

## 方案 C：GPU 程序化形变

最终方案仍是：

```text
静态局部顶点和法线
+
每只怪物的姿态参数
+
Vertex Shader 程序化变形
```

GPU 同时变换 Position 和 Normal，然后在 Shader 内计算分档明暗。

这才适合：

```text
220 只完整怪物
60 FPS
动态腿部动画
无距离 LOD
```

# 还有一个单纯的配色问题

你的基础颜色本身就是偏紫灰色：

```text
Leg     = RGB(84, 75, 96)
Foot    = RGB(72, 64, 82)
Abdomen = RGB(102, 92, 112)
Thorax  = RGB(90, 80, 101)
```

因为蓝色通道始终最高，而且地面是暗绿色，互补对比会让紫色更加明显。

再乘上最低明暗 `0.64` 后，腿部大约变成：

```text
RGB(54, 48, 61)
```

在当前暗绿色环境中很容易接近黑色。

最高档 `0.88` 则大约是：

```text
RGB(74, 66, 84)
```

所以仅四档明暗就足以让同一批蜘蛛呈现明显的浅紫与黑紫差异。

建议先将明暗档收窄：

```ts
const MATTE_SHADE_LEVELS = Object.freeze([
  0.76,
  0.8,
  0.84,
  0.88,
] as const);
```

并把基础色稍微中性化，例如：

```text
Leg     = RGB(76, 76, 82)
Foot    = RGB(66, 67, 72)
Abdomen = RGB(92, 91, 98)
Thorax  = RGB(81, 80, 88)
```

不过，**先修复姿态与烘焙明暗不同步，再调色**。否则调完颜色以后，蜘蛛依然会随着历史更新时机出现不一致。

当前最应该改的是：

```text
临时验证：
关闭 CPU 世界方向烘焙明暗

近期正式方案：
固定 Facet Shade + 事件驱动基础色

长期方案：
GPU 形变 Position/Normal + Shader 分档光照
```
